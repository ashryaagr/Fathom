# Why this spec exists — quoted user instruction (verbatim, do not paraphrase)

> "We can have another tab. Inside, whenever we have a research paper, we start indexing the first time. Once the indexing is done, we also start making a high-level diagram of the paper that describes the entire workflow. That workflow is described using Excalidraw. We integrate Excalidraw as a sub-tab. In the top left on the bar or tab appears the whiteboard that Excalidraw creates for the summary of the paper.
>
> We can obviously iterate over that by having a side chat on the right. It is stored as an .excalidraw file, but its goal is to help me understand the core parts of the paper. There can be multiple diagrams in that same whiteboard, but it's designed to help me understand the core pieces. Hence we need to have a very careful research-backed strategy on how we want to do this. What is the best way to prompt the model so that it generates the right explanation? We need to have a grounded, multi-stage pipeline or workflow that ends up synthesizing the information that ultimately gets fed into an Excalidraw whiteboard diagram. There needs to be careful planning. It is a separate module that needs to be built.
>
> The Excalidraw should use Excalidraw casual, handwritten fonts.
>
> It should explain to me different parts of the paper and only the main parts. Sometimes I don't care about literature; I care about the core methodology of the paper, and if there are any core algorithms, then perhaps we can zoom in on that. The whiteboard helps me explain at different levels. The first diagram explains to me at a high level what even happens in the paper. We then start to zoom in on the next diagram, and then we zoom in on the different parts, and so on. We zoom in until we achieve a level of detail that is good and understandable enough, and the user can choose to stop at whatever level of zoom they want. We should do at least two or three levels of zoom, and we need to think carefully about how the zooming in, etc., should work."

# What is being built

A **Whiteboard tab** alongside the existing PDF view. After the paper finishes indexing, the AI generates an Excalidraw whiteboard that explains the paper at multiple zoom levels of abstraction — Level 1 (what the paper does), Level 2 (each major part), Level 3 (algorithms / components within those parts). The user navigates the zoom levels at their own pace and stops when comprehension is sufficient. A side chat lets them ask follow-ups and request refinements; the chat updates the diagram. The whiteboard persists as an .excalidraw file alongside the paper's index.

This is **not** a TOC, **not** a section outline, **not** a literature map. It is a *visual mental model* of the paper's core methodology + algorithms, generated by the AI from the indexed paper, refined by user dialogue.

## Hard requirements (taken directly from the user's instruction)

1. **A separate tab** (top-left tab bar or similar) — distinct from the PDF view, accessible without leaving the paper.
2. **Triggered after indexing completes** — first-open of a paper kicks off the whiteboard generation as part of the existing index pipeline, not on user demand.
3. **Excalidraw as the rendering engine** — must be the actual Excalidraw component, not a re-implementation. Casual / handwritten fonts (Excalifont, hand-drawn style).
4. **Multi-level zoom** — at least 2–3 levels. Level 1 = "what even happens in the paper" (high-level workflow). Level 2 = each independent part. Level 3 = algorithms / components within a part. User chooses where to stop.
5. **Multiple diagrams in the same whiteboard** — not a single canvas; a series of related diagrams the user navigates between.
6. **Side chat on the right** — iterative refinement. Chat history is part of the whiteboard's saved state.
7. **Persisted as an .excalidraw file** — alongside the paper's index folder, survives reload, portable.
8. **Core methodology + algorithms ONLY** — not literature, not related-work, not citations. The user is explicit: "I don't care about literature; I care about the core methodology."
9. **Grounded, multi-stage pipeline** — the AI must not hallucinate structure that isn't in the paper. Multi-stage prompting required.
10. **Research-backed prompting strategy** — not vibes-prompting. AI Scientist agent is researching the right techniques.
11. **Careful planning before build** — this is a separate module, not an iteration on the lens. PM, AI Scientist, Cog Reviewer all sign off before implementation.

## Explicit non-goals (also from the user's instruction)

- **No literature / citation mapping.** Use Connected Papers / ResearchRabbit if you want that — Fathom doesn't.
- **No section TOC.** PDF outlines already do that; nobody needs another one.
- **No auto-quiz or self-explanation prompts** in the whiteboard. (The earlier `structural-reading.md` proposal floated this; it's out of scope here.)
- **Not a replacement for the PDF.** The whiteboard *complements* the source; the user still reads the paper. The whiteboard is the mental-model scaffold they build in parallel.

# Design questions the research agents are addressing

These are the open questions the in-flight research must answer before this spec can be marked implementable.

## 1. AI Scientist research — pipeline & prompting

- **What's the multi-stage pipeline that gets from indexed paper → Level 1 diagram → Level 2 diagram → Level 3 diagram?** Single LLM pass per level vs. extraction → abstraction → diagram-DSL → Excalidraw vs. something else.
- **How do we ground the model so each diagram is faithful to the paper?** What grounding inputs does each level need (full text, abstract+figures, parent diagram, etc.)?
- **How do we control level of detail at each zoom?** What does "Level 2" mean operationally — extract-N-stages vs. force-N-nodes vs. let the model decide?
- **How do we keep diagrams visually consistent between zoom levels** so the user feels they're zooming into the same thing, not seeing 3 unrelated artworks?
- **What's the right diagram DSL or intermediate format?** Direct Excalidraw JSON generation vs. Mermaid → convert vs. node/edge list → layout engine vs. structured natural language → renderer.
- **How do we handle iteration via the side chat?** Does the chat regenerate the whole diagram, patch it, or open a sub-conversation for the diagram-edit?
- **What grounding evidence do we cite to the user?** Each node in the diagram should link back to the source paragraph/figure it came from (so they can verify and dive in).
- **What's the prompt-caching strategy** to keep this affordable when the user iterates frequently?

## 2. Visual-abstraction researcher — diagram design at each zoom

- **What does Level 1 ("what even happens in the paper") look like?** Single canonical pattern (input → process → output) or paper-type-dependent? How many nodes max? How much text per node?
- **What does Level 2 look like** when zooming into a Level 1 node? Same visual grammar or a new one?
- **What does Level 3 look like for an algorithm?** Pseudocode-shaped vs. flowchart vs. state diagram?
- **How do papers themselves illustrate their workflow** in their own Figure 1? What's the convergent visual grammar across ML / systems / theory papers?
- **What sketchnoting / visual-thinking research is applicable?** (Visual Thinking Strategies, Sunni Brown's *Doodle Revolution*, Dan Roam's *Back of the Napkin*.)
- **What's the right way to indicate "you can zoom into this"** vs "this is a leaf node"? Affordances.
- **How should the user navigate between zoom levels** — keyboard / pinch / sidebar / breadcrumb? Must feel continuous with Fathom's existing dive gesture.
- **Where do the source-paragraph citations live** in the visual? Tooltip on hover? Faint marker beside the node?

# Pipeline (v2 — LLM-friendly, 2-pass)

User direction (2026-04-25, verbatim): *"I don't want to do a multi-stage pass. As a matter of fact, two passes are more than okay. Try to use as much context as possible for stage one so that the model is able to identify the important parts that it wants to select, because Opus 4.7 has a 1 million context window and we want to use that. Once we extract the important parts, we can synthesize from stage two whatever you want, because synthesizing some good summaries, et cetera, from the important parts in stage two should keep as minimal as possible. Design all of this to be LLM friendly as much as possible. Enforcing rigorous structures can often be counterproductive when working with agents."*

```
[Index complete]
       ↓
[Pass 1 — UNDERSTAND]                                          [Pass 2 — RENDER]
   Opus 4.7 (1M context)                                          Sonnet 4.6
   Input: ENTIRE paper                                            Input: cached Pass 1 output
          (content.md + figure descriptions                              + render request
           + digest.json + optional purpose anchor)              Output: WBDiagram for one
   Tools: none                                          ────►            specific frame
   Output: structured-but-loose markdown                        Cached:  Pass 1 output
            "Understanding doc"                                  Called:  1× Level 1 (eager)
   Cached:  this output                                                  N× Level 2 (lazy)
       ↓
       [Understanding doc]
       (markdown sections: Goal, Methodology, Components, L1 plan, L2 plans)

                                                                       ↓
                                                                [WBDiagram JSON]
                                                                       ↓
                                                  [ELK.js auto-layout → Excalidraw scene]
                                                  (deterministic TS, no LLM)
```

## Pass 1 — UNDERSTAND

| Field | Value |
|---|---|
| Model | **Opus 4.7** (chosen for the 1M context window — user's explicit pick). Pilot option: Sonnet 4.6 with 1M-beta — if pilot quality matches Opus on 5 sample papers, drop to Sonnet to cut cost ~70% (AI Scientist note, non-blocking). |
| Tools | **`Grep` on `content.md` only.** No `Read` (already loaded), no `WebSearch`, no `Bash`. The Grep escape hatch lets Opus self-verify "did the paper actually say X?" before committing it to the understanding doc — Chain-of-Verification pattern (Dhuliawala et al. 2023), strictly improves faithfulness with low latency cost. AI Scientist's blocking revision; absorbed. |
| Input | System prompt + ENTIRE `content.md` + figure-caption text + `digest.json` + optional purpose anchor. **Capped at ~80k tokens** (long-context attention degrades on multi-hop synthesis past that — RULER benchmark, Hsieh et al. 2024). For longer papers/surveys: chunk by section using `digest.json`, run Pass 1 per super-section, merge in a thin synthesis step. |
| Output | A markdown document — Opus's understanding of the paper, organised into the sections it finds natural. Suggested but **not enforced** structure: `## Goal`, `## Core methodology`, `## Components`, `## Suggested Level 1 diagram`, `## Suggested Level 2 expansions` |
| Structure | Markdown with H2 sections. **No rigid JSON schema, no Zod-validated leaves, no mandatory `evidence:[{page,quote}]` on every component.** The model writes naturally; we parse what we can. |
| Citations | The model is **encouraged** to quote source paragraphs with page tags inline (`[p.4]`) so the renderer can attach citations to nodes — but quotes are not required on every leaf. Better to have the model write fewer, accurate citations than to force one per node and get filler. |
| Cached | Yes — full output marked `cache_control: ephemeral`. Reused for every Pass 2 call. **Implementation must enable 1-hour cache TTL beta** — default 5-min TTL would force every side-chat patch beyond 5 min to pay full uncached input, breaking the $0.005/patch economics. |

### Pass 1 prompt template (the actual prompt — addresses AI Scientist's blocking gap)

```
You are reading a research paper to help a curious technical reader build a mental
model of what the paper does — its core methodology, its key components, and how
they fit together. NOT literature review, NOT related work, NOT acknowledgements.

You have:
  • The full paper text (with `<!-- PAGE N -->` markers).
  • Figure captions and references.
  • A pre-computed digest (sections, equations, glossary).
  • Optional: the user's stated purpose for reading this paper. If present, weight
    your synthesis toward what they came to learn — but don't omit core methodology
    they didn't explicitly ask about.

You may call `Grep` on `content.md` if you want to verify a specific phrase before
committing it to your output. Use it sparingly — the paper is already in your
context. Grep is for confirming verbatim quotes, not for re-reading.

Reason section by section first (out loud, in <thinking> tags if you support them,
otherwise as a numbered list). Then synthesise.

Output a markdown document organised however feels natural. As a STARTING TEMPLATE
(not a contract — adapt freely):

## Goal
One sentence: what does this paper do? Then one paragraph of context: why does it
matter, what problem space does it sit in.

## Core methodology
The heart of the paper, in plain language. 2–4 paragraphs. Reference figure
numbers when figures carry the explanation (e.g. "see Figure 2"). Quote sparingly
when the paper's wording is load-bearing — use inline `[p.N]` page tags so a
downstream renderer can attach citations.

## Components
The 4–7 distinct pieces that make up the methodology. For each: one-line label, a
2-3 sentence description, source page reference, and (if applicable) the figure
that depicts it. These are the candidate nodes for the Level 1 diagram.

## Suggested Level 1 diagram
The user will see a top-level diagram with at most 5 nodes (working memory limit).
If your Components list above has more, propose a grouping — which Components
collapse into which Level 1 nodes. Use the paper's own vocabulary.

## Suggested Level 2 expansions
For each Level 1 node that contains 2+ Components, briefly say what its zoom-in
should show. Skip Level 1 nodes that are leaf concepts.

Hard rules:
  - Do NOT invent components, stages, or relationships that aren't in the paper.
    If you're not sure, say so.
  - Use the paper's own terminology. Don't rename things.
  - If the paper isn't a methods/system/algorithm paper (e.g. a theory paper or a
    survey), say so explicitly at the top and adapt the structure: theory papers
    show theorems instead of stages; surveys show the categorisation taxonomy.
  - Quote inline (`[p.N]: "..."`) only when confident the quote is verbatim. The
    downstream renderer will run a soft verifier and flag any unverified quotes
    with a `?` marker — you don't need to be perfect, but flagged citations
    look bad to the user, so quote conservatively.
```

**Why this shape:** the user's explicit guidance is to use as much context as possible and let the model do the selection. Loading the whole paper into Opus 4.7's 1M window means we never lose information at extraction time the way a multi-pass narrowing pipeline does. The model decides what's important — not a Zod schema. This is the "LLM-friendly" framing the user asked for.

## Pass 2 — RENDER (called per diagram)

| Field | Value |
|---|---|
| Model | **Sonnet 4.6** (constrained generation; cheaper; sufficient since the cognitive heavy-lifting was Pass 1) |
| Tools | None |
| Input | System prompt + cached Pass 1 output + render request: either *"Render Level 1"* or *"Render Level 2 for the node labelled `<X>` in the Level 1 plan"* |
| Output | `WBDiagram` JSON — node/edge list with optional citations and optional summaries. Renderer is tolerant: missing fields default; extra fields ignored; layout-hint optional. |
| Cached | The Pass 1 output is the cached prefix. Pass 2 calls cost only the small render-request delta + small JSON output. |

**WBDiagram (loose schema, all fields optional except `nodes` and `edges`):**

```jsonc
{
  "level": 1,                                  // 1 or 2
  "title": "Title at top of diagram",          // optional
  "parent": "L1.2",                            // present on Level 2 only
  "nodes": [
    {
      "id": "L1.1",
      "label": "Token Embed",                  // ≤ 4 words, ≤ 24 chars
      "kind": "input"|"process"|"output"|"data"|"model",  // tolerant: unknown → "process"
      "summary": "...",                        // optional, ≤ 25 words
      "drillable": true,                       // defaults to false
      "citation": { "page": 4, "quote": "..." } // optional
    }
  ],
  "edges": [
    { "from": "L1.1", "to": "L1.2", "label": "..." }   // label optional
  ],
  "layout_hint": "lr" | "tb"                   // defaults to "lr"
}
```

The renderer (deterministic TS) maps `kind` → palette, runs ELK.js for hierarchical auto-layout, and emits Excalidraw scene JSON with proper bindings + customData carrying citations. Nothing in this DSL is a hard constraint that breaks the model — missing summaries, missing citations, unknown kinds all degrade gracefully.

## Anti-hallucination — soft, not structural

Original pipeline auto-dropped any node whose quote didn't grep-verify against `content.md`. **v2 changes this to a soft check:**

- After Pass 1, run a quote-verifier as a *background* pass over every inline `[p.N]`-tagged quote. Verifier normalises whitespace + lowercase + strips punctuation, then accepts ≥85% trigram overlap as "verified." Hard "unverified" only when overlap is <50% with anything in `content.md`. (AI Scientist note: the strict-byte-match in v1 was over-tight; most real failures are paraphrase or whitespace, not fabrication.)
- Verifier results are logged to `whiteboard-issues.json` for inspection. The diagram is NOT mutated based on verifier output.
- The understanding doc is left intact; the renderer attaches the citation with a **two-channel verified/unverified affordance** (cog reviewer §6 required fix — single `?` glyph alone fails redundancy for color-blind/low-vision):
  - **Verified citation marker**: solid amber square, no glyph. (Default.)
  - **Unverified citation marker**: dashed amber square outline + faint `?` glyph in the corner. The dashed border is the shape signal; the `?` is the redundant glyph signal. Either alone is sufficient to read "unverified" without colour.
- Banner threshold is **data-driven, not the round number**. Default to 40% in the first build; instrument per-paper verification rate; after observing 20 real papers, calibrate. The cog reviewer flagged that a banner firing on every 4th paper habituates within 2 weeks (Bravo-Lillo et al. 2013) — better to err on the side of not banner'ing than to wallpaper the warning.

Rationale: the user told us "rigorous structures can often be counterproductive." Forcibly dropping nodes whose citations don't lex-match the source is exactly that — it hides the model's actual reasoning behind a strict pre-filter. Better to surface unverified citations as such and let the user judge.

## Pass 1 streaming UX (cog reviewer non-blocking note)

The 60–70s Pass 1 latency is Doherty-acknowledged within 1 frame (skeleton + placeholder outlines + spinner), but the cognitive *feel* between v1 (three perceptible "things landed" moments) and v2 (~50s of nothing visible, then Level 1 hydrates) is worse despite being technically compliant.

**Mitigation**: if the Opus streaming API exposes incremental tokens during Pass 1, render them into a small auto-scrolling sidebar — even raw markdown, even partial — so the user has *something* to read while waiting. Strictly informational; collapses to nothing once Pass 2 hydrates the diagram. This is how the existing lens streaming works (see `LensCard.tsx`); same pattern applies here.

## Cost-tier option (cog reviewer non-blocking note)

The opt-in consent prompt currently reads *"Generate whiteboard for this paper? · ~$1.50 · ~60 s"*. Johnson & Goldstein 2003 + cog reviewer's read of consent UX: a 4× cost jump from v1's $0.40 will shift acceptance rate downward, not just consent quality. **Defer building** a Sonnet-Lite alternative — but instrument first-paper acceptance rates after launch. If acceptance drops below ~60% in dogfood, add a *"Lite (~$0.50, Sonnet only)"* toggle next to the Generate button. Don't pre-build.

# Cognitive design principles already known

These come from CLAUDE.md §1 and the cog-review skill, applied to this feature:

- **Reduce cognitive fatigue is the only principle that ultimately matters** (CLAUDE.md §1). Diagrams that look impressive but don't lower the reader's load are failures.
- **Element interactivity, not element count, drives cost** (CLAUDE.md §1; Sweller, Chen & Kalyuga 2010). Each diagram should keep the count of *interacting* elements at ≤ 4±1 per zoom level (Cowan 2001 working-memory chunk size). A Level 1 diagram with 12 nodes is broken even if it's "complete."
- **Scaffolding must be dismissable** (CLAUDE.md §1; Kalyuga 2003). The whiteboard tab is opt-in / closeable. Experts who don't need it shouldn't see it consume screen space.
- **Recursion has one visual grammar** (CLAUDE.md §2.1). Zooming from Level 1 → Level 2 must feel like the existing dive gesture — same gesture, same animation, same back-navigation. The whiteboard zoom and the lens drill share visual grammar, not invent a new one.
- **The reader controls the resolution** (CLAUDE.md §1). The pipeline pre-generates as deep as the user pulls; it does not push detail.
- **Apple-level smoothness** (CLAUDE.md §2.2). Zooming between levels must be a smooth transition, not a tab-switch.

# Visual + interaction design (resolved — full sketches in `whiteboard-diagrams-research-visual-abstraction.md`)

Headline decisions from the Visual Abstraction researcher; full sketches of Level 1/2/3 for ML / systems / theory papers, plus the prior-art justification, live in the sidecar.

- **Single Level 1 grammar across paper types**: input → pipeline → output, with a star/badge on the novel block. Survey of Transformer / ResNet / MapReduce / AlphaGo / BERT / Diffusion / PRIMES Figure 1s shows convergence on this template — paper type changes labels, not shape.
- **Hard ceiling: 5 nodes per diagram (Cowan 4±1)**. Cog reviewer veto if exceeded. If the paper has more stages, the AI groups them; the grouped node becomes drillable.
- **Level 2 = same canvas, animated zoom (NOT new tab).** Excalidraw `scrollToContent` with 320 ms `cubic-bezier(0.4, 0, 0.2, 1)` (inside Doherty's 400 ms). Parent's stroke + label persist as a soft frame around the L2 subgraph; sibling L1 nodes ghost to ~20% at canvas edges. The user feels they are *inside* the parent, not next to it. Same recursion grammar as the existing PDF dive (CLAUDE.md §2.1).
- **Level 3 = napkin card** (Roam, *Back of the Napkin*): hand-drawn rectangle with 5–8 lines of pseudocode in Excalifont + one inline mini-diagram on the right showing the shape transformation. Three templates: pseudocode card / flowchart / state diagram, picked by what the algorithm IS (sequential / branching / state).
- **Navigation**: ⌘+pinch on a node = drill in (matches existing dive). Esc / two-finger swipe right = drill out (matches existing lens history). Persistent breadcrumb top-left in system sans (`Paper ▸ Encoder ▸ Self-Attention`). NO mini-map (would split foveal attention).
- **Drillable vs. leaf**: dashed inner border + ⌖ glyph for drillable; solid border, no glyph for leaf; dashed border + spinning ⌖ for generating. Three non-colour signals (CLAUDE.md §6 colour-blindness rule).
- **Source citations**: amber square in each node's top-right corner (same amber as PDF lens markers — one unified marker grammar). Hover = quote tooltip. Click = jump to PDF + pulse the source paragraph. ⌘+click = open a Fathom lens on that paragraph.
- **Excalidraw integration**: `@excalidraw/excalidraw` React package; per-node metadata in `customData`; multi-diagram via Excalidraw's native `frame` element (one frame per zoom level, all in one `.excalidraw` file). Specific gotchas (Excalifont registration, keyboard-focus theft, view-mode while streaming) documented in the sidecar.
- **Side chat**: 320 px right rail, shrinks the canvas (does not overlay). 4 controls only (Hick's Law): chat history, Ask box, "Apply to canvas", "Reset diagram". Scoped per-frame (Level 1 of paper vs Level 2 of Encoder are separate threads).
- **Doherty acknowledgement contract** (cog reviewer §3): every user-initiated whiteboard transition must produce a visual response within ONE frame (≤ 400 ms, target ≤ 50 ms). Specifically:
  - **First click on the Whiteboard tab**: within 1 frame, render the breadcrumb skeleton + 5 placeholder node outlines + a "Generating…" glyph in the canvas. Real nodes hydrate as the stream lands (S1 ~25–40 s, S2 ~6 s, S3 ~3 s).
  - **Click on a drillable node (Level 2 drill-in)**: within 50 ms, the parent-frame outline begins drawing AND the spinning ⌖ glyph appears on the clicked node. Even if the L2 generation hasn't started yet (cold cache), the user sees acknowledgement.
  - **Side-chat patch submission**: within 50 ms, the affected node(s) get a soft outline pulse so the user knows the patch was received before the diff renders.
  - The 45 s first-paint and ~5 s per-iteration latencies are fine *as long as the immediate ack is in*. Implementation must not let any user-visible interaction wait for a network call before painting confirmation.
- **Generation consent — opt-in per paper, not auto-on after indexing** (cog reviewer §8): the prior recommendation to silently auto-generate the whiteboard at index time was rejected on consent grounds. The user pays for the API spend (~$0.40/paper) via their own Claude CLI auth (CLAUDE.md §7), so silently spending on every paper they open is a financial-consent violation, not just a UX one (Johnson & Goldstein 2003 — defaults stick for ~85% of users). Resolution:
  - First time the user clicks the Whiteboard tab on a given paper, the canvas shows a one-tap affordance: *"Generate whiteboard for this paper? · ~$0.40 · ~45 s"*. No modal, no popup — just an inline button inside the tab the user already opened.
  - On accept, the pipeline runs; preference is remembered per-account as `auto_generate_whiteboard_on_index = true` so subsequent papers index-and-generate together.
  - The setting is reversible from Preferences. Re-flipping to false reverts to the per-paper consent prompt.

First Whiteboard-tab click for a paper shows a one-tap "Generate · ~$0.40 · ~45 s" inline button; accept once, the pipeline runs; user can flip a Preferences toggle to "auto-generate on index" if they want first-use friction removed for future papers.
